### Finding 1 — [MEDIUM] Restore of a non-root deleted block can create a live orphan under a still-deleted parent
**Verdict: CONFIRMED** (severity MEDIUM appropriate; one of the two posited reach-paths is dead, but the other is live via pure UI)

**Evidence checked**:
- `commands/blocks/crud.rs:1037-1083` — `restore_block_inner` validates only existence,
  deleted-ness, and `deleted_at == deleted_at_ref`. No parent/ancestor-liveness check. The
  CTE (`descendants_cte_standard!`) walks *downward* only. Confirmed exactly as described.
- `loro/projection.rs:491-525` — `project_restore_block_to_sql` has the same downward-only
  shape (uses `descendants_cte_cohort!`). Confirmed.
- `pagination/trash.rs:8-89` — **decisive**: the trash list returns **only roots** of each
  deletion batch (root = `parent_id IS NULL OR parent.deleted_at != b.deleted_at`).
- `pagination/hierarchy.rs:66` — `list_children ... WHERE parent_id IS ?1 AND deleted_at IS NULL`,
  so a live block under a deleted parent is invisible in the tree. Orphan-invisibility holds.
- `mcp/tools_rw/` — no production MCP `restore_block` write tool (only `#[cfg(test)]`
  references in `tools_rw/tests.rs`). `soft_delete/restore.rs:13-22` is documented test/bench-only.

**Adjudication of the two reach-paths the report posits**:
- *"caller passes a non-root descendant id directly"* → **NOT reachable**. Trash UI only
  surfaces roots; no production MCP restore tool exists. A synced `RestoreBlock` op only ever
  carries a root id (it is generated by the same root-only paths). So the projection-path
  "reproduce on every device" claim is overstated for descendant ids.
- *"ancestor was deleted in a different cascade/cohort"* → **IS reachable via pure UI**.
  Concretely: child B soft-deleted at T1; later parent P soft-deleted at T2. The cascade
  filters `deleted_at IS NULL`, so deleting P does NOT re-stamp the already-deleted B — B
  keeps T1. Because `B.parent (P, T2) != B.deleted_at (T1)`, B qualifies as a **trash root**
  (`trash.rs:62-68`) and the FE offers it for restore. Restoring B sets `B.deleted_at = NULL`
  while P stays deleted → live block under a deleted, invisible parent. The invariant
  "a live block's ancestors are live" is broken, and B is absent from both tree and trash.

This is precisely the cross-cohort hazard the codebase already invested in elsewhere (#1119,
cohort-keyed restore to avoid over-restore). The *symmetric* gap — under-restore leaving an
orphan — is not guarded. No test asserts parent-liveness on restore (checked
`restore_cascade_tests.rs`, `delete_restore_convergence_tests.rs`, crud.rs guards at
:1041-1059). Genuine, unhandled, reachable. MEDIUM stands.

**BETTER-APPROACH**: The report's two-option fix is right; prefer the *restore-upward* option
(restore the chain up to the nearest live ancestor) over hard `InvalidOperation`, because
rejecting would leave the user with a block they can see in trash but can never restore.
Apply in BOTH `restore_block_inner` and `project_restore_block_to_sql` (and the test-only
`soft_delete/restore.rs`) so the command path and replay/sql_only path converge. The
`tracing::warn!` breadcrumb is a good cheap floor if the full fix is deferred.

### Finding 2 — [MEDIUM] FTS rank keyset epsilon band disagrees with exact ORDER BY (skip/dup across page boundary)
**Verdict: CONFIRMED-BUT-RESEVERITY → LOW**

**Evidence checked**:
- `fts/search/fetch.rs:150-152` (keyset WHERE) and `:265` (`ORDER BY fts.rank, b.id`, exact,
  no epsilon). `:266` `LIMIT ?5`.
- `fts/search/cursor.rs:107-167` — cursor stores the **exact** `last.search_rank` (f64) and
  `last.id` of the previous page's last row; resume binds them as `?3`/`?4`.

**Logic re-derived and confirmed**: cursor = `(X, C)`. Correct next page = rows sorting after
`(X,C)` by exact `(rank,id)`. A row R with `X < rank_R <= X+eps` and `id_R <= C` *should* be
on the next page (rank strictly greater ⇒ sorts after, regardless of id) but: clause-1
`rank > X+eps` is false; clause-2 needs `id > C` which is false ⇒ **R is silently skipped**.
Symmetrically a row with `X-eps <= rank_R < X` and `id_R > C` is admitted though it sorts
before the cursor ⇒ **duplicate**. So the report's mechanism is correct, and `b.id` tiebreak
is NOT sufficient — the band groups *unequal* near-ranks under the id tiebreak while the sort
treats them as ordered. The "equal-rank rows disambiguated by block_id" doc claim only covers
*exactly*-equal ranks (which ARE handled correctly).

**Why re-severity to LOW**: the bug only bites when a page boundary falls between two rows
whose bm25 ranks differ by a *nonzero* amount < 1e-9 relative. Exactly-equal ranks are fine;
differences ≥ eps are fine. bm25 over distinct documents almost never lands in the open
(0, 1e-9·|X|) gap at the precise `LIMIT` boundary. No panic, no corruption — at worst one
result missing/repeated on a "Load More" in a pathological near-identical-content corpus.
That is a textbook LOW edge, not MEDIUM. The report itself concedes "real-world loss is
unlikely."

**BETTER-APPROACH**: Agree with the report's "make both ends consistent." Simplest correct
fix: keep the exact `(rank, id)` keyset in the WHERE (drop the epsilon band) — `last.search_rank`
is the *same f64 SQLite produced for that row*, round-tripped through the cursor, so re-feeding
it to an exact `rank > X OR (rank = X AND id > C)` keyset has no float-drift problem in
practice (the drift the epsilon guards against, per #1598, is largely theoretical here since
the value originates from SQLite, not a recomputation). If the authors want to keep absorbing
drift, snap rank to a quantized integer key used identically in WHERE and ORDER BY. Note this
is a deliberate trade-off the authors made (#1598) — file as a low-priority consistency
cleanup, not a defect.

### Finding 3 — [LOW] Unchecked `n * 7` in weekly recurrence shift
**Verdict: EXAGGERATED → keep LOW, but trim impact framing; borderline TRIVIAL**

**Evidence checked**:
- `recurrence/parser.rs:103` `let n: i64 = num_str.parse().ok()?;`, `:108` `if n <= 0 { return None; }`
  (no upper bound), `:113` `"w" => base + chrono::Duration::days(n * 7)`, vs `:119-120`
  `"m" => shift_by_months(base, n)?` / `"y" => ... n.checked_mul(12)?`. The asymmetry is
  exactly as described — `w` is the lone unchecked multiply among the m/y siblings.

**Why EXAGGERATED**: the framing ("debug panic / release wrap") is technically true but the
report under-states that the *adjacent* `"d" => base + chrono::Duration::days(n)` arm is
*also* unguarded — `chrono::Duration::days` itself multiplies by 86_400_000 internally and
panics/saturates on extreme `n` regardless of the `*7`. So "the `w` arm is the lone unchecked
multiply" is not quite the whole picture: making only `w` checked still leaves `d` (and even
checked `w`) able to overflow inside `Duration::days`. The real defect is narrower (just the
explicit `n*7` arithmetic) AND the real fix is broader (bound `n` at parse time, which also
covers `d`). Reachable only via an ~18-digit hand-typed/imported interval string — single-user
threat model, no malicious actor, so worst realistic case is a debug-build panic the developer
sees, or a wrong date the user immediately notices on an absurd input.

**BETTER-APPROACH**: Don't just swap in `checked_mul(7)?` (the report's fix) — that leaves the
`d` arm and the `Duration::days` internal multiply exposed. Instead cap/validate `n` once at
parse time (`:103-110`), e.g. reject `n` above a sane interval ceiling, which fixes all four
arms uniformly and is simpler than per-arm checked math. This is a free, correct hardening but
is genuinely TRIVIAL given the threat model — synthesizer may reasonably fold it into a
"recurrence input bounds" cleanup rather than a standalone issue.
