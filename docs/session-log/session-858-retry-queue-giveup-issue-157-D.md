**Summary:** Closes sub-item **D** of issue #157 — the `materializer_retry_queue` had no give-up cap, so a permanently-failing op (e.g. the 2026-05-19 spike where a remote op kept trying to splice into a locally-missing block) would loop every 60 s for days, producing ~3 log lines per attempt. The fix wires two give-up triggers into the existing sweep path (`sweep_once` in `src-tauri/src/materializer/retry_queue.rs`):

- `attempts >= MAX_ATTEMPTS` (10) — bounds the *active* failure case. The 2026-05-19 spike pattern.
- `created_at < now - GIVE_UP_AGE_DAYS` (7) — bounds the *stale* case where a row's been sitting in the queue beyond the keep-trying window even if its retry count is still low.

When either trigger fires, the sweeper logs a warn line with `give_up_reason=max_attempts` / `give_up_reason=age_exceeded`, increments the new `retry_queue_giveup_total` metric, and deletes the row. No further retries.

No migration needed — `materializer_retry_queue.attempts` and `.created_at` columns already exist (migration 0044). The change is purely additive: a new constant pair, a `give_up_reason()` helper, a check at the top of the `sweep_once` per-row loop, a new field on `QueueMetrics` + `StatusInfo`, and 3 tests.

**Process notes:** `MAX_ATTEMPTS = 10` and `GIVE_UP_AGE_DAYS = 7` are taken verbatim from #157's table-row "Predicate" column. They're constants in `retry_queue.rs` rather than configurable via env / settings because (a) #157 doesn't ask for a knob, (b) the right tuning signal is the `retry_queue_giveup_total` metric — if operators see it climbing in normal use, the constants are wrong and the right response is to fix the failing task class rather than relax the give-up cap, and (c) keeping them as compile-time constants pins the schema-test snapshots and avoids runtime-config drift.

**Lessons learned (for future sessions):** When threading a new metric through `QueueMetrics → coordinator::status_with_scheduler → StatusInfo`, the insta snapshot at `commands/tests/snapshots/...snapshot_status_info_response.snap` is the regression net — accept the `.snap.new` after diffing to confirm only the intended field appears (and at the intended position; the JSON ordering matches the struct field order). The specta bindings.ts regen step is similarly mechanical via `cargo test -- specta_tests --ignored` and must happen before the pre-commit hook's `ts_bindings_up_to_date` check runs.
