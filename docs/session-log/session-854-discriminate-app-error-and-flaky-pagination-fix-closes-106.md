**Summary:** Closes #106 by adding three discriminated `AppError` paths so the frontend can distinguish recoverable / expected sqlx outcomes from generic database errors at the IPC boundary. Backend adds two new variants — `AppError::PoolTimedOut` (unit; IPC `kind: "pool_busy"`) and `AppError::Conflict(String)` (unique-constraint message; IPC `kind: "conflict"`) — and reuses the pre-existing `AppError::NotFound(String)` for `sqlx::Error::RowNotFound`, avoiding a 40-site churn that a new unit variant would have caused. Translation happens once in a manual `impl From<sqlx::Error>`, not at each call site; the generic `Database(sqlx::Error)` variant stays as the catch-all. Frontend exposes typed predicates `isNotFound` / `isPoolBusy` / `isConflict` / `isDatabaseError` plus a shared `retryOnPoolBusy(thunk, opts)` helper (default schedule `[50ms, 150ms]`) wired into autosave, form-submit conflict surfaces (TagList create + rename, PageBrowser create), and the alias-resolver silent-empty-state path.

**Also fixed in this PR (user-requested):** five flaky pagination integration tests that were riding the post-MAINT-112 race where every `create_block_inner` enqueues a background materializer dispatch, even for content-type blocks. The tests' helper docstrings still claimed "content-type creates dispatch NO background materializer tasks" — stale since MAINT-112 — and the tests relied on that assumption by skipping `settle_bg_tasks`/`settle`. Observed reproducible flake rate ~12% in isolation; ~30% in full-suite runs. Added `settle_bg_tasks(&mat).await` / `settle(&mat).await` after the seed loop and before pagination in each, plus updated the helper docstrings on `create_content` and `settle_bg_tasks` in `integration_tests.rs` to reflect MAINT-112's actual semantics. Stress-tested 30 isolated runs and one full-suite run: zero flakes, zero failures.

**Process notes:** Two parallel build subagents (backend + frontend) converged independently on reusing `AppError::NotFound` for `RowNotFound` — both spotted the existing variant via codebase grep before writing anything. The flaky-test fix was added mid-iteration after the user noticed the flaky tag still appeared in nextest output on the previous PR and asked to bundle the fix in. The root cause was a stale docstring claiming content-type creates don't dispatch bg tasks; MAINT-112 (which switched all writes to `CommandTx::commit_and_dispatch`) invalidated that, but the test helper documentation was never updated.

**Lessons learned (for future sessions):** When a refactor changes the dispatch model for a whole class of operations (here MAINT-112 → bg dispatch on every block op), grep test-helper docstrings that reference the old model and update them in the same PR. Otherwise the stale docs misinform test-writers for months, and the resulting flake rate is low enough to be tolerated rather than tracked down.
