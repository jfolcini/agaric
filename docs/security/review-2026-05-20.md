## Part 1 — STRIDE walkthrough

Each boundary's row in the threat model was re-read against the current code. "Still load-bearing?" answers below; rationale only where the answer is anything other than yes.

### B1 — Frontend ↔ Backend (Tauri IPC)

| STRIDE | Mitigation cited in threat model | Still load-bearing? | Notes |
|--------|----------------------------------|---------------------|-------|
| Spoofing | Capability allowlist (`src-tauri/capabilities/default.json`) | Yes | Verified ~100 `tauri::command` handlers across `src-tauri/src/commands/`; every one is `pub` and registered through the typed handler list. |
| Tampering | sqlx `query!` macro + parameter binding | Yes | All `query!`/`query_as!` macros bind parameters; no string concatenation of user input. The `tauri command sanitize` prek hook (`prek.toml` line 386) blocks new handlers that bypass the `AppError` path. |
| Repudiation | Append-only op log (`op_log` table) | Yes | Every materialiser-routed write produces an `op_log` row; the op-log-immutability invariant is enforced in `materializer.rs`. |
| Info-disclosure | No native filesystem read/write outside `app_data_dir()` + capability list | Yes | Tauri 2.11.2 capability model unchanged; no permissive `path:*` capability granted. |
| DoS | Read pool capped at 4 conns; write pool single-conn; `BEGIN IMMEDIATE` on write tx | Yes | The search-cancellation work (shipped in this main) adds `tokio::sync::watch` cancellation to the search read path so a typing burst can't pin all 4 read conns. |
| Elevation | `tauri::Builder::invoke_handler!` is the only IPC entry | Yes | Inspected `src-tauri/src/lib.rs` (`invoke_handler` registration around line 1322) — every registered command is in `src-tauri/src/commands/` and routes through `AppError`. |

**Findings:** none. The B1 contract is unchanged in substance since the threat model was written; only the cancellation row's "no per-request cancellation" caveat has been removed (the cancellation work closed it).

## Part 2 — Diff-sweep

Since this is the first review, the "since prior review" baseline is project inception. The sweep walks the in-scope directories' major commits.

### IPC boundary (`src-tauri/src/commands/`)

~100 handlers across 19 files at HEAD. Major change vectors since project start:

- **Search-stack work** — All new IPC handlers added (`search_blocks`, `search_blocks_partitioned`, `list_tags_by_prefix`, etc.) go through `AppError`, sanitiser, and the `cargo nextest run` happy + error-path test gates. The search-cancellation work added `tokio::sync::watch` cancellation propagation through the search builder; no privilege change.
- **Compound-filter Phase 1 (FilterPrimitive extraction)** — not yet shipped as of this review; outside scope.
- **MCP rmcp adapter** — rmcp adapter is now the sole MCP server (commits `60c15329`, `5fd225d2`). MCP is exposed via Unix domain socket bound to a per-session path with restrictive file mode; out-of-process; no IPC privilege escalation. ✅
- **Spaces work** — every command that accepts a `space_id` validates it through `space_filter_clause!` macro before touching SQL. Reviewed `properties.rs`, `pages.rs`, `queries.rs`; no SQL injection vectors. ✅

**Findings:** none.

### Sync transport (`src-tauri/src/sync_*`)

No protocol-level changes since project start beyond the iroh-spike work (out of scope; that spike has its own threat model when it lands as iroh transport adoption (#78)). The TLS + TOFU + CBOR frame shape is unchanged.

**Findings:** none.

### OAuth (gcal command + gcal_push modules — removed 2026-06-15)

A `REVIEW-LATER.md` row notes a deprecated `tauri-plugin-shell::open` callback at `gcal.rs:550-557`, behind `#[allow(deprecated)]`. The deprecation is a UI-routing concern, not a security one (it opens the user's default browser to the OAuth consent URL). Tracked, not security-blocking.

The search-audit follow-up work shipped in this main does not touch `gcal.rs`.

**Findings:** none.

### Updater (`tauri-plugin-updater = "2.10.0"`)

Agaric uses the upstream `tauri-plugin-updater` crate; there is no per-app updater Rust file. The plugin is wired in `src-tauri/src/lib.rs` and configured via `src-tauri/tauri.conf.json`. The Tauri 2.10.3 → 2.11.2 bump (commit `eb128ebe`) closed CVE-2026-42184 in the Tauri stack; no further updater changes since. Threat-model row B4 covers the current shape correctly.

**Findings:** none on the Rust side. CI-side findings see B4 above.

### MCP exposure (`src-tauri/src/mcp/`)

Major refactor since project start: spike → production flip → drop of the legacy wire path (commits `54df262b`, `d3a0953a`, `a4637abe`, `27d74238`, `5fd225d2`, `60c15329`). Inspected:

- Socket binding (`mcp/socket.rs`) — binds to a per-session path under `XDG_RUNTIME_DIR` with mode `0600`. Verified the mode-setting code; no world-readable variant.
- Tool dispatch (`mcp/rmcp_adapter.rs`) — every exposed tool routes through `AppError`. The dispatch table is statically generated; no eval-style code paths.
- Authentication — the MCP socket is single-user-only (no auth needed; the socket file's permissions ARE the auth). The threat model's "single-user trust model" out-of-scope row covers this.

**Findings:** none. The rmcp adapter has more code than the legacy path but the trust boundary is identical.
