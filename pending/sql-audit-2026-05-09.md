# SQL audit — 2026-05-09

Deep review of all SQL and its invocations across `src-tauri/` and `src/`.
Focus: performance, maintainability, correctness — pushing work into the
database wherever possible. Target scale: 100k pages / millions of blocks.

**Method:** five parallel Explore agents (schema/indexes, N+1/fetch_all,
pagination/limits, filter-sort-group-in-code, transactions/concurrency),
followed by a verification pass that read every cited `file:line` driving
a HIGH or CRITICAL claim. Hallucinations and exaggerations were discarded
and are listed at the bottom for transparency.

**Cost legend:** S = ≤½ day, M = 1–3 days, L = >3 days.

---

## Headline assessment

The codebase shows mature SQLite engineering. Transactions are correct
(`BEGIN IMMEDIATE`, post-commit dispatch wrapper, `INSERT OR IGNORE` for
op_log idempotency, `foreign_keys=ON`, WAL, `query_only=ON` on the reader
pool). Pagination is overwhelmingly keyset — only one OFFSET in the
codebase, deliberately bounded to depth 1000 in undo and documented.
Recent index hygiene migrations (0045, 0048-0050) show real understanding
of the planner.

The real defects cluster in two places:

1. **The backlink/filter pipeline materializes large sets in Rust before
   paginating** — a popular page with 50k backlinks loads all 50k IDs
   into a hash set before slicing.
2. **The agenda view ships rows to the frontend and then intersects /
   sorts / groups in JS** — this is exactly the anti-pattern the audit
   was asked to flag.

Plus a handful of schema and PRAGMA hardening items.

## CRITICAL & HIGH findings (verified)

### H2. `PropertyNum` and `PropertyDate` filters fetch all matching rows then filter in Rust

**File:** `src-tauri/src/backlink/filters.rs:176-204` (Num), `:206-235` (Date)

Both load every row with the property key set, then apply the comparison
operator (`<`, `<=`, `=` with `f64::EPSILON`, etc.) in Rust. The sibling
`PropertyText` (lines 135-174) was already fixed in PERF-27 — Num/Date
were missed.

There is also a latent correctness issue: `(v - value).abs() < f64::EPSILON`
for `Eq` is much stricter than SQLite's `=` and may return a different
match set.

- **Risk:** Low. Inline the operator into the `WHERE` clause as PropertyText
  already does.
- **Impact:** HIGH for users with numeric/date properties — a `score < 5`
  filter against 100k blocks materializes the full set.
- **Cost:** S. ~2 hours, mirroring the PropertyText shape.

### H3. Frontend agenda intersects filter results in JS after fan-out IPCs

**Files:** `src/lib/agenda-filters.ts:417-477`, `:26` (`AGENDA_QUERY_LIMIT = 500`)

`executeAgendaFilters` issues one IPC per sub-filter, each capped at 500
rows, then computes the intersection in JS at line 477:

```ts
blocks = [...intersection].map((id) => allBlocks.get(id) as BlockRow).filter(Boolean)
```

This silently drops any block outside the top-500 of any sub-query —
precisely the regression that motivated the new backend
`filtered_blocks_query_inner` (`src-tauri/src/commands/queries.rs:1067-1078`,
comment: *"Replaces the FE pattern of fan-out IPCs … fixes the silent-cap
regression"*). The backend fix exists and is wired up; the frontend has
not migrated to use it.

- **Risk:** Low. The replacement command is already implemented and tested.
- **Impact:** CORRECTNESS bug at small scale, plus performance/scale benefit.
  At 100k pages this is the difference between "right answer in one query"
  and "wrong, truncated answer assembled in JS".
- **Cost:** S–M. Replace the multi-invoke + intersect path in
  `agenda-filters.ts` with a single `filtered_blocks_query` invocation.

### H4. Frontend agenda sorts and groups in JS

**File:** `src/lib/agenda-sort.ts` (420 lines, sort sites at lines 30, 108,
175, 230, 242, 265, 325, 331, 339, 354)

`sortAgendaBlocks` re-sorts blocks by `(effective_date, state_rank,
priority_rank)` in JS; `groupBy*` functions then re-group and re-sort
within groups. Backend can already do all of this in a single query:
`blocks` has `due_date`, `scheduled_date`, `todo_state`, `priority` as
columns (migration 0012/0013), so

```sql
ORDER BY COALESCE(due_date, scheduled_date),
         CASE todo_state WHEN 'DOING' THEN 0 WHEN 'TODO' THEN 1 ... END,
         CASE priority   WHEN 'A' THEN 0 ... END
```

is straightforward.

- **Risk:** Medium — the JS sort/group logic is intricate (state/priority
  enums with custom rank). A SQL implementation must match exactly to
  avoid UI churn.
- **Impact:** MEDIUM-HIGH. `AGENDA_QUERY_LIMIT=500` masks the issue today,
  but raising that limit (or removing it via H3) immediately exposes JS
  sort cost.
- **Cost:** M. One ORDER BY per sort key; group breaks can be derived in
  SQL via window functions or computed in a single linear JS pass over
  a pre-sorted result.

### H6. Missing performance PRAGMAs (`cache_size`, `mmap_size`, `temp_store`)

**File:** `src-tauri/src/db.rs:378-393`

`base_connect_options` sets `journal_mode=WAL`, `synchronous=NORMAL`,
`foreign_keys=ON`, `wal_autocheckpoint=5000`, `journal_size_limit=52MB`,
`busy_timeout=5s`. **Not set:** `cache_size` (default ≈ 2000 pages = 8MB),
`mmap_size` (default 0), `temp_store` (default = file).

For a 100k-page database the on-disk size is easily several hundred MB.
With the default 8MB page cache the working set is constantly thrashing.

- **Risk:** Low. These are well-understood, well-documented PRAGMAs.
- **Impact:** HIGH. Empirically, a bigger cache (`PRAGMA cache_size = -65536`
  for 64MB) and a memory-mapped read region (`PRAGMA mmap_size = 268435456`
  for 256MB) cut hot-query latency by 2–10× on multi-hundred-MB SQLite
  databases. `temp_store=MEMORY` avoids spilling temp B-trees during
  large sorts.
- **Cost:** S. Three lines in `base_connect_options`. Worth A/B
  benchmarking on a real 100k-page corpus before committing the values.

### H7. Foreign keys lack `ON DELETE` behavior on most core tables

**Files:** `src-tauri/migrations/0001_initial.sql:9, 14, 19-20, 26, 31, 37-38, 45, 95, 102, 109` and 0015, 0021, 0025

`block_tags`, `block_properties`, `block_links`, `attachments`, `tags_cache`,
`pages_cache`, `agenda_cache`, `page_aliases`, `block_tag_inherited`,
`projected_agenda_cache` all `REFERENCES blocks(id)` without `ON DELETE`
clauses. Newer tables (`block_tag_refs` 0034, `block_drafts` 0038)
correctly use `ON DELETE CASCADE`, so the inconsistency is real.

With `PRAGMA foreign_keys=ON` (good — confirmed line 384), a hard
`DELETE FROM blocks WHERE id = ?` fails if any referencing row exists.
The codebase compensates with explicit cascade logic in
`soft_delete/trash.rs::cascade_soft_delete`, but the schema doesn't
enforce it. Soft-delete leaves dangling cache rows until the next
cache rebuild.

- **Risk:** Medium. Adding CASCADE to an existing FK requires the table-
  rebuild dance in SQLite (`CREATE TABLE _new + INSERT SELECT + DROP +
  RENAME`). For `block_tags` / `block_properties` that touches the entire
  user database — must run inside the migration transaction.
- **Impact:** MEDIUM. Defense-in-depth correctness; fewer "stale cache row
  pointing at a deleted block" classes of bugs.
- **Cost:** M. One migration per table; recommend doing all in one batch
  with a single transaction. Test on a populated database first to
  confirm migration runtime is acceptable.

## MEDIUM findings (verified)

### M3. `flush_all_drafts_inner` reads every draft into memory at boot

**File:** `src-tauri/src/commands/drafts.rs:168-171`

```sql
SELECT block_id, content FROM block_drafts ORDER BY updated_at ASC
```

with no LIMIT. The table is `(block_id PRIMARY KEY, content, updated_at)`
so it's bounded by the number of blocks with unflushed drafts, but a
pathological state (10k unflushed drafts) loads everything at once.

- **Risk:** Low — well-defined contract that this path is "all-or-nothing"
  by design.
- **Impact:** LOW practically; MEDIUM in degenerate cases.
- **Cost:** S. Either chunk by `LIMIT N` and run multiple transactions,
  or add a hard cap with a warning when truncated.

## Hallucinations / exaggerations from the initial pass (excluded)

For transparency. These were claimed by subagents but didn't survive
verification:

1. **"`block_tag_inherited` is missing an index on `block_id`"** — FALSE.
   The PRIMARY KEY is `(block_id, tag_id)`, which gives an implicit
   B-tree on `block_id` for free
   (`migrations/0021_block_tag_inherited.sql:11`). Same for several
   other "missing index" claims that overlooked PK / UNIQUE constraints —
   `idx_page_aliases_page` and `idx_agenda_date` were *deliberately
   dropped* in 0045 for exactly that reason.
2. **"Redundant `resolve_root_pages` call at `backlink/grouped.rs:102-117`"**
   — FALSE. The target block being viewed is the page itself; `base_ids`
   contains *source* blocks linking to it, never the target. The two
   calls resolve different block sets.
3. **"Backlink grouped re-sorts three times"** — exaggeration. Step 9
   (`grouped.rs:282-287`) is a positional `sort_by_key(|&(_, pos)| pos)`
   to align rows with the already-sorted global order — O(N) on the
   small per-group bucket, not a real sort.
4. **"Pagination audit found no frontend filtering"** — incomplete.
   `agenda-filters.ts:477` and `agenda-sort.ts` (entire file) are
   exactly the anti-pattern the user wanted flagged. The pagination
   agent missed this; H3 and H4 above correct the omission.
